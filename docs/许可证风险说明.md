# 许可证风险说明

## 1. 结论先行

本项目采用“只研究概念、独立设计、独立实现”的洁净实现策略：不复制四个参考项目的代码、测试、数据库迁移、接口定义、页面组件、CSS、图片、图标、文案或部署脚本；不把参考项目作为依赖或子模块；不改变既定技术栈。

这比逐项判断代码片段是否构成衍生作品更稳妥，尤其适用于 OpenMES 的 AGPL-3.0 和 ERPNext 的 GPL-3.0。本文是工程风险说明，不构成法律意见；正式对外交付、商业许可或 SaaS 上线前应由律师复核。

## 3. 各许可证的主要工程风险

### 3.1 OpenMES：AGPL-3.0

AGPL 是强 copyleft 许可证，并特别覆盖通过网络与用户交互的修改版本。若将其代码复制、修改、紧密结合后用于 Web 系统，可能产生向网络用户提供相应源代码的义务；若对外分发，还涉及许可证文本、版权声明和完整相应源码等要求。

本项目是 Web MES，正处在 AGPL 高敏感使用场景，因此：

- 不复制 Laravel 模型、服务、控制器、React/Inertia/Livewire 页面、测试和迁移。
- 不根据源码逐行翻译成 Python/TypeScript；换语言并不会自动消除衍生风险。
- 不复刻具有高度具体表达的页面布局、组件层级、CSS 和图标组合。
- 可以借鉴生产计划、步骤执行、Andon、角色范围等抽象思想，并用本项目术语重新建模。

### 3.2 ERPNext：GPL-3.0

GPL 同样是强 copyleft 许可证。内部纯使用与向客户交付/分发的义务不同，但本项目不应把商业交付模式建立在这种边界判断上。复制 ERPNext Python/JavaScript/JSON DocType、迁移或报表并分发，可能要求整个结合作品满足 GPL 条件。

因此只使用 ERPNext 官方文档帮助理解制造领域对象。Work Order、Job Card、Routing、Workstation 等是通用领域概念，但其具体字段集合、校验流程、源代码和页面表达不复制。

### 3.3 frePPLe：MIT 社区版与专有企业版

MIT 对代码复用较宽松，但通常要求在复制或重要部分中保留版权与许可声明。frePPLe 又采用社区版/企业版双许可，官方文档可能同时描述两种版本的能力。

风险点：

- 看到公开文档不等于对应功能源代码属于 MIT。
- 计划编辑、库存计划、预测、企业集成等能力应逐项对照当前[功能表](https://frepple.com/docs/current/features.html)与仓库目录。
- 企业版代码、构建产物、截图和专有文档内容不能当成 MIT 资产。
- 本项目即使理论上可复用社区 MIT 代码，也按本次要求全部自行实现，以避免混入 C++/Django 技术栈和许可证通知管理负担。

### 3.4 yuwang/MES：README 声明 MIT，但许可证正文缺失

仓库 README 的许可证章节写有 “MIT License”，但核对版本没有独立许可证全文，也没有清晰的版权通知文件。这说明作者表达了 MIT 意向，却使需要保留的具体版权和许可文本不完整。

在维护者补充标准 LICENSE 文件或作出明确书面确认前，不应把任何源码或页面资产复制到本项目。只研究其轻量分层、工单和报工的抽象思路。

## 4. “概念借鉴”与“代码复制”的边界

### 4.1 允许的研究成果

- 用自己的语言描述领域对象职责和业务状态。
- 比较“模板—快照—实例”“需求—供应分配”“有限资源日历”等通用设计思想。
- 基于福兰特 Excel 样表和业务规则重新命名实体、字段、接口和页面。
- 从空白文件开始，用既定技术栈编写自己的算法、SQLAlchemy 模型、FastAPI 服务和 Vue 组件。
- 链接官方文档和仓库作为研究来源。

### 4.2 禁止的行为

- 复制或轻微改写源文件、函数、测试用例、SQL、迁移、配置和脚本。
- 将 PHP/Python/C#/JavaScript 逐行翻译为本项目 Python/TypeScript。
- 保留参考项目特有的类名、接口路径、异常文案和非常规字段组合，只改变量名。
- 从页面截图反向复刻布局尺寸、配色、组件结构和 CSS。
- 复制商标、Logo、图标、插图、示例数据或文档大段文字。
- 引入 AGPL/GPL 服务作为隐藏的 Docker Compose 组件而不做许可证评估。
- 把 frePPLe Enterprise/Cloud 功能误记为社区 MIT 能力。

## 5. 洁净实现流程

### 5.1 研究阶段

1. 研究人员只产出概念说明、差异、业务规则和来源链接，不保存源代码片段到项目仓库。
2. 每条采纳建议必须能用福兰特业务理由解释，例如“防止模板变化污染历史工单”，不能以“某项目这样写”为唯一理由。
3. 对特有交互只写用户目标和状态变化，不写原组件树或 CSS 参数。

### 5.2 实现阶段

1. 开发人员以本仓库需求文档、Excel 分析和五份研究结论为输入，从空白代码实现。
2. PR 模板增加许可证确认：是否查看/复制第三方代码、是否新增依赖、依赖许可证是什么。
3. 使用依赖扫描和 SBOM 记录 Python、npm、Docker 基础镜像许可证；这与四个参考项目的研究隔离开。
4. 发现类似实现时优先重写为本项目业务术语、测试和数据结构；无法确认来源时停止合并并复核。

### 5.3 证据留存

建议维护 `docs/third-party-research-register.md` 或等价登记表，记录：来源、commit、访问日期、许可证、研究主题、是否引用、实现负责人和复核人。本文可作为第一条研究记录。

## 6. 依赖与部署注意事项

本项目固定栈自身也需要持续许可证治理：

- Python/npm 依赖只从官方包管理器引入，锁定版本并保留 lockfile。
- Docker 镜像、数据库驱动、甘特图库、Excel 组件和图标库必须逐项核对许可证。
- Element Plus 本身不意味着所有第三方甘特组件均可商用；排产时间轴应优先自主实现或选择许可证明确的依赖。
- 若未来把 frePPLe 作为独立外部 APS 服务接入，仍需重新评估网络/API 结合、部署包、企业许可和客户交付边界；本次架构不包含该依赖。
- 客户私有化部署与 SaaS 运营分别进行许可证清单复核，AGPL 尤其不能只按“未分发”简单判断。
