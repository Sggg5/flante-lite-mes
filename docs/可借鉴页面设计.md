# 可借鉴页面设计

## 1. 页面设计目标

页面不是复刻 OpenMES、ERPNext、frePPLe 或 yuwang/MES 的布局。以下设计基于它们揭示的工作方式，再针对福兰特现有 Excel 流程重新组织，全部由 Vue 3、TypeScript 和 Element Plus 自主实现。

三条体验原则：

1. 计划员始终看得见“还有多少没排、为什么不能排、拖动后影响谁”。
2. 班组长始终看得见“现在做什么、前序是否就绪、还差多少、出现问题如何上报”。
3. 管理者看到的颜色都能点开数值证据和处理记录，不能只有红黄绿。

## 2. 信息架构

| 一级导航 | 页面 | 主要角色 |
|---|---|---|
| 首页 | 生产总览、异常摘要 | 管理者、计划员、班组长 |
| 数据准备 | 导入中心、批次详情、映射与错误处理 | 计划员、管理员 |
| 补库 | 补库运行、建议审核、计算证据 | 计划员、审核人 |
| 生产需求 | 需求池、需求详情 | 计划员 |
| 生产计划 | 周排产中心、计划版本、发布校验 | 计划员、管理者 |
| 车间执行 | 我的工序任务、任务执行、快速报工 | 班组长、操作者 |
| 异常 | 异常中心、异常详情 | 全部授权角色 |
| 基础资料 | 产品、工艺、设备、模具、产能、日历 | 工艺员、管理员 |
| 系统 | 用户、角色、数据范围、审计日志 | 管理员 |

## 3. 周排产中心

OpenMES 最值得借鉴的是待排池与时间轴合在一个工作台，而不是让计划员在多个表单间来回跳转。frePPLe 最值得借鉴的是每次移动都重新给出可行性；ERPNext 则提醒我们加入工作站日历和能力。

### 3.1 推荐布局

```text
┌ 周期 2026-07-13~07-19  工序[全部] 版本[V3草稿]  搜索...  [校验] [保存] [发布] ┐
├───────────────┬───────────────────────────────────────────────────────────┤
│ 待排需求池     │ 周一      周二      周三      周四      周五      周六      周日 │
│ 筛选/排序      ├───────────────────────────────────────────────────────────┤
│ □ 紧急  □异常 │ 制管01  [任务A────────]       [任务B────]                  │
│ D-102 余1200  │ 制管02       [任务C──────]                                 │
│ D-108 余 800  │ 下料组  [任务D──] [任务E────]                             │
│ D-115 余 600  │ 成型05                [任务F────────]                     │
│ ...           │ 包装组                         [任务G──────]              │
├───────────────┴───────────────────────────────────────────────────────────┤
│ 影响检查：设备负荷 92% | 模具冲突 1 | 前序不足 2 | 交期风险 3  [查看明细]       │
└───────────────────────────────────────────────────────────────────────────┘
```

页面区域：

- 顶部：周范围、工序/设备/产品族筛选、草稿/发布版本、撤销/重做、校验和发布。
- 左侧待排池：需求号、产品、剩余待排、优先级、交期、连续未排天数、工艺缺失提示。
- 中部时间轴：按工序→工作中心→设备分组；支持日/班次粒度、拖放、拉伸、拆分和换资源。
- 底部影响栏：每次编辑后的容量、前序、模具、交期和需求占用影响。
- 右侧详情抽屉：任务与需求来源、能力依据、前序供给、修改原因和审计链。

### 3.2 拖放交互

1. 从待排池拖入资源行时，先显示幽灵条和建议起止时间，不立即写数据库。
2. 前端提交产品、数量、资源和时间；后端返回标准工时、冲突列表、警告和受影响任务。
3. 无阻止项时保存草稿；有警告时要求选择原因；有阻止项时禁止，除非用户有 `plan.override_conflict`。
4. 拖动已冻结任务时先提示“该任务已发布/进入冻结窗口”，不能无声重排。
5. 一个需求被部分拖入后，左侧剩余量立刻更新；仍大于 0 时继续留在池中。

### 3.3 冲突表现

| 类型 | 表现 | 默认动作 |
|---|---|---|
| 需求超分配 | 红色边框、显示超出数量 | 阻止保存 |
| 设备/模具重叠 | 条形叠层阴影，列出另一任务 | 阻止；授权覆盖需原因 |
| 日能力超载 | 资源行负荷条变橙/红并显示分钟缺口 | 警告或阻止，按配置 |
| 前序不足 | 任务左侧断链图标，显示可用量缺口 | 警告；发布前阻止 |
| 工艺缺失 | 待排卡片禁拖并给出缺失步骤 | 阻止 |
| 交期风险 | 任务尾部显示延迟天数 | 警告 |
| 冻结任务受影响 | 锁图标和受影响任务清单 | 阻止或高级授权 |

颜色旁必须有图标、文字和数字，照顾色弱用户，也避免误判。

### 3.4 发布与版本

“发布”打开校验抽屉，按阻止、警告、提示分组；必须填写发布说明。发布成功显示版本号、发布人和时间。插单、批量顺延、换设备会生成新草稿版本，并提供版本差异页：新增、移除、数量变化、日期变化、资源变化和原因。

不采用 frePPLe 的全库场景复制，也不采用 ERPNext 的通用文档提交界面；本项目只维护周计划业务版本。

## 4. 车间操作与报工

OpenMES 的现场页面强调大按钮、步骤顺序和异常一键上报；ERPNext Job Card 强调人员、工位、开始/完成和时间日志；yuwang/MES 说明简单数量表单有较低学习成本。本项目组合这些原则，但不复制页面。

### 4.1 我的任务列表

默认按当前用户授权工序过滤，适配 10~13 英寸平板：

```text
┌ 我的任务  工序：成型  日期：今天  班次：白班  [扫码/搜索] ┐
│ [进行中 3] [待开始 8] [阻塞 1] [已完成 16]              │
├───────────────────────────────────────────────────────────┤
│ 紧急  T-260717-018  DN25弯头  批次 B260713  设备 CF-05   │
│ 计划 1200 | 合格 650 | 不良 18 | 剩余 532  前序：已就绪   │
│ 08:00—14:00               [继续生产] [报工] [异常]        │
├───────────────────────────────────────────────────────────┤
│ T-260717-021 ... 前序可用 300 / 本次计划 600              │
│                               [查看] [异常]                │
└───────────────────────────────────────────────────────────┘
```

任务卡只展示现场需要的字段：任务、产品/规格、批次、设备/模具、计划与剩余、前序就绪、优先级和异常。完整来源与版本放进详情，不把计划员表格搬到现场。

### 4.2 任务执行页

顶部固定显示任务身份和阻塞状态；中部按工艺顺序展示当前步骤、前序和下一步；底部固定三个主要动作：开始/暂停、报工、报告异常。

动作规则：

- 未 READY 时“开始”禁用，并直接说明缺少前序、工单未释放或有阻塞异常。
- 开始动作确认人员、设备、模具和班次；已有进行中记录时直接继续。
- 报工弹窗预填任务、设备、班次和实际开始时间，数字输入使用大号步进器和数字键盘。
- 完工不能只靠按钮；提交报工后由有效数量和必需步骤状态派生。

### 4.3 报工表单

| 区域 | 字段/交互 |
|---|---|
| 任务摘要 | 计划数量、累计合格/不合格/报废、预计剩余；只读 |
| 本次数量 | 合格、不合格、报废；均为 Decimal；不能同时为 0 |
| 时间 | 实际开始/结束，默认带入，可授权修正 |
| 未完成 | 本次未完成原因：换模、缺料、设备、质量、人员、插单、其他 |
| 异常 | 可勾选“同时创建异常”，选择等级并写说明/上传图片（图片后续阶段） |
| 提交 | 防重复点击；成功显示报工号和更新后的累计/剩余 |

不能提供“直接编辑/删除历史报工”。报错后在报工详情执行“申请冲销/更正”，显示原记录、反向记录、原因、操作者和审批信息。

## 5. 生产需求池

需求池是本项目防漏排的核心页面，不应藏在工单列表里。

### 5.1 主表

固定列：需求号、产品编码、名称、规格、来源、确认量、有效分配量、最终合格量、剩余待排、剩余待完成、交期、连续未排天数、状态和异常。

默认视图优先显示：

1. 已确认但完全未排；
2. 部分排产且跨周；
3. 已排但逾期/无报工；
4. 正常进行中；
5. 已完成/已关闭。

点击数量打开证据抽屉：确认量来自哪次建议审核、分配到哪些工单、哪些分配已释放、最终合格量来自哪些有效报工。

### 5.2 详情页

使用时间线串联：补库运行 → 建议审核 → 需求建立 → 分配/释放 → 计划版本 → 工单/工序任务 → 报工/冲销 → 异常/解决。ERPNext 的来源追踪和 frePPLe 的 pegging 思想在此以轻量方式落地。

## 6. 补库建议中心

页面保留用户熟悉的 Excel 横向数据感，但不照搬单元格格式：

- 第一组固定列：产品、规格、产品族、常规/订单生产。
- 第二组销量：近六个月逐月、最大、六个月平均、三个月平均、加权结果。
- 第三组供给：库存、预计入/出、管材在制、管件在制、已排未开工。
- 第四组结果：算法、目标库存、系统建议、人工确认、审核状态和修改原因。

提供“查看计算证据”抽屉，展示公式、输入批次、原始值、归一值和每一步结果。批量修改算法/确认量前显示影响条数和总数量；审核和转需求均要求幂等且提供结果摘要。

20,000 SKU 场景必须服务端分页、筛选、排序和虚拟化；产品选择器不能一次加载全部选项。

## 7. 异常中心

### 7.1 列表与聚合

顶部按严重度、规则、责任工序和状态聚合；列表显示异常号、等级、类型、对象、数值证据、持续时间、建议动作、责任范围、状态和最近发现时间。

推荐异常类型首版：

- 确认需求未排、少排、连续跨周未排；
- 计划超产能、设备/模具冲突、前序不足；
- 已到计划时间未开始、进行中无报工、计划逾期；
- 未完成未结转、负在制、产品/工艺/产能数据缺失；
- 报工超量或数量投影不一致。

### 7.2 处理流程

OPEN → ACKNOWLEDGED → RESOLVED → CLOSED。确认时指定责任人/范围，解决时填写措施和证据，关闭时复核。自动扫描发现相同 `dedupe_key` 时更新最近发现时间，不重复刷屏；已解决问题重现时重新打开并保留历史。

## 8. 工艺、工作站和能力页面

参考 ERPNext 的 Routing/Workstation 对象划分，但使用本项目字段：

- 工艺模板列表：编码、版本、状态、生效期、绑定产品数、修改人；发布后只读。
- 路线编辑器：左侧工序库，右侧纵向步骤；拖动只改变顺序，不是甘特图；配置必需、可重叠、损耗率、转移批量和可用工作中心。
- 产品工艺页：当前绑定、历史版本、未来生效版本。
- 设备/模具日历：周视图显示可用班次、停机、维修。
- 产品能力矩阵：产品 × 工序 × 设备/模具，服务端搜索和批量导入。

## 9. 首页看板

首页不追求大屏装饰，优先回答五个问题：

1. 还有多少确认需求未排？
2. 本周各工序负荷是否超限？
3. 本周计划完成到哪里？
4. 哪些任务已逾期或停止报工？
5. 哪些异常需要当前用户处理？

指标卡：待排需求数量/条数、部分排产、连续两周未排、本周计划量、最终合格量、完成率、阻塞任务、开放异常。所有卡片都可带筛选跳转明细。

## 10. 权限和审计页面

- 角色页以权限树配置“模块—对象—动作”，例如计划查看/编辑/发布/冲突覆盖、报工创建/冲销、异常确认/解决。
- 用户页除角色外配置车间、工序和工作中心数据范围。
- 审计页按用户、时间、对象、动作和请求 ID 查询，左右对比 before/after；业务原因单独突出。
- 计划版本和报工冲销应在各自业务详情展示，不强迫用户到通用审计页寻找。

## 11. 响应式与性能要求

- 计划中心以 1440px 及以上桌面为主；车间页以平板横竖屏和触控为主。
- 长表使用固定列、列分组、服务端分页/排序和按需虚拟滚动。
- 甘特只加载当前可视周和资源组，拖动采用乐观视觉反馈，保存结果以服务端校验为准。
- 所有写操作显示处理中并使用幂等键；冲突响应保留用户未保存的草稿。
- 不在前端重算权威数量、状态或能力，前端只做即时预览。

## 12. 明确不照搬的页面做法

- 不复制 OpenMES 的组件、样式、图标组合、文案或前后端接口。
- 不复制 ERPNext 的 DocType 表单和完整制造单据字段。
- 不复制 frePPLe 的计划编辑器或依赖其企业版界面。
- 不复制 yuwang/MES 的 React/Ant Design 页面，也不采用可直接编辑删除报工的交互。
- 不为了“像专业 APS”在首版加入自动优化按钮；先让每个警告可解释、每次人工调整可追溯。
