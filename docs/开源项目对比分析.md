# 开源制造管理项目对比分析

## 1. 研究范围与结论

本文用于“福兰特轻量生产计划执行系统”的设计研究，不是代码选型或移植方案。研究时间为 2026-07-17，核对的仓库版本如下：

| 项目 | 核对版本 | 技术与定位 | 许可证结论 |
|---|---|---|---|
| [Mes-Open/OpenMes](https://github.com/Mes-Open/OpenMes/tree/49373668f8990bec69758c636f3b2ffaebcf7556) | `49373668`，v0.16.2 | Laravel/Inertia/React 等构成的 MES，计划与现场执行体验较完整 | [AGPL-3.0](https://github.com/Mes-Open/OpenMes/blob/49373668f8990bec69758c636f3b2ffaebcf7556/LICENSE) |
| [frappe/erpnext](https://github.com/frappe/erpnext/tree/9fdb6dd87562ff7b9e28e4c3d944c469a77a187a) | `9fdb6dd8`，develop | 完整 ERP；制造模块以单据、库存和成本闭环为核心 | [GPL-3.0](https://github.com/frappe/erpnext/blob/9fdb6dd87562ff7b9e28e4c3d944c469a77a187a/license.txt) |
| [frePPLe/frepple](https://github.com/frePPLe/frepple/tree/16a88c8362cde70d30ead700669483c8f1839fb6) | `16a88c83`，master | 需求预测、库存计划和有限产能 APS | [社区版 MIT、企业版专有双许可证](https://github.com/frePPLe/frepple/blob/16a88c8362cde70d30ead700669483c8f1839fb6/COPYING) |
| [yuwang/MES](https://github.com/yuwang/MES/tree/876234f5131a9bb74f36e5c38c216583a8eb45d0) | `876234f5`，main | .NET/React/MySQL 的轻量 MES 示例 | [README 声明 MIT](https://github.com/yuwang/MES/blob/876234f5131a9bb74f36e5c38c216583a8eb45d0/README.md)，但该版本没有独立 LICENSE 正文 |

总判断：

- 页面交互优先借鉴 OpenMES，制造对象边界优先借鉴 ERPNext，计划思想优先借鉴 frePPLe，工程分层尺度参考 yuwang/MES。
- 本项目不能变成 ERP 或 APS。首要目标仍是把“Excel 补库 → 生产需求 → 周排产 → 工序任务 → 报工 → 异常闭环”做清楚。
- 四个项目均只提炼领域概念和交互原则。本项目继续使用 Vue 3、TypeScript、Element Plus、FastAPI、SQLAlchemy、PostgreSQL、openpyxl、pandas 和 Docker Compose，全部代码自行实现。
- frePPLe 社区代码虽然是 MIT，也不直接复制；OpenMES/ERPNext 的强 copyleft 代码和界面实现更不得复制；yuwang/MES 在许可证正文缺失未澄清前同样不复制。

## 2. 总体对比

| 维度 | OpenMES | ERPNext | frePPLe | yuwang/MES | 本项目取舍 |
|---|---|---|---|---|---|
| 产品主数据 | 轻，产品类型绑定工艺 | 重，Item 覆盖采购、库存、销售、制造 | 计划型 Item，带层级与 MTS/MTO | 极简编码、名称、规格、启停 | 保留轻量主档，补产品族、别名、单位、补库策略、工艺版本 |
| 生产需求 | 工单为核心，独立需求层较弱 | 销售订单/物料请求汇入 Production Plan | Demand 是计划根对象，支持交期、优先级和交付策略 | 无独立需求对象 | 独立 `production_demand`，必须保存来源和分配链 |
| 工单/任务 | Work Order → Batch → Batch Step | Work Order → Operation → Job Card | Manufacturing Order/OperationPlan | WorkOrder → WorkReport | 计划任务与工序执行任务分开，但不引入完整 ERP 工单体系 |
| 工艺路线 | 模板版本 + 工单 JSON 快照 + 步骤实例 | Routing/BOM Operation + Workstation + Job Card | Routing/alternate/split operation | 未实现，仅预留 | 关系型模板版本 + 发布时快照 + 任务实例 |
| 周排产 | 拖拽、缩放、待排池、冲突强制覆盖 | 生产计划、能力计划、日历，交互偏单据 | 求解计划 + 可行性 Gantt 调整 | 未实现 | OpenMES 式周界面 + frePPLe 式容量/可行性提示；首版不做求解器 |
| 状态 | 计划、阻塞、暂停、完成等执行态 | 单据态与库存/工序进度共同派生 | proposed/approved/confirmed/completed/closed | 四态自动流转 | 状态由动作与数量派生，不允许前端自由改枚举值 |
| 报工 | 操作者按步骤开始/完成/报异常 | Job Card 计时、人员、完成量、损耗 | 执行回写不是社区版核心强项 | 表单录入良品/不良品 | 大按钮快速报工；追加写、冲销，不物理改删历史 |
| 数量 | planned/produced，批次汇总 | qty、produced、process loss、operation completed | quantity、completed、pegging、库存投影 | target、completed、defect 增量汇总 | 明确确认量、有效分配、合格、不合格、报废及两个“剩余”口径 |
| 异常 | Andon Issue，可阻塞工单 | 通知、校验、缺料/超产限制 | 标准化 Problem Report 和库存状态 | 设备报警；业务异常较弱 | 规则引擎式 `data_issue`，去重、责任、处理闭环 |
| 权限 | Admin/Supervisor/Operator + 产线范围 | 角色、DocType、权限级别、用户权限 | Django 用户/组/场景权限 | 固定角色 + 控制器授权 | RBAC + 工序/车间数据范围 + 动作权限 |
| 审计 | 前后值、用户、IP、UA，强调不可变 | Version 和 Audit Trail | 变更历史、场景隔离 | 有日志实体但使用不完整 | 追加写审计、业务变更原因、计划版本、报工冲销 |

## 3. OpenMES

研究依据包括项目 [功能说明](https://github.com/Mes-Open/OpenMes/blob/49373668f8990bec69758c636f3b2ffaebcf7556/README.md)、[WorkOrder 模型](https://github.com/Mes-Open/OpenMes/blob/49373668f8990bec69758c636f3b2ffaebcf7556/app/Models/WorkOrder.php)、[BatchStep 模型](https://github.com/Mes-Open/OpenMes/blob/49373668f8990bec69758c636f3b2ffaebcf7556/app/Models/BatchStep.php)和[生产计划页面](https://github.com/Mes-Open/OpenMes/blob/49373668f8990bec69758c636f3b2ffaebcf7556/resources/js/Pages/ProductionPlanner/Planner.jsx)。

### 3.1 产品主数据结构

`ProductType` 以编码、名称、描述、单位、启用状态为核心，关联产线、工艺模板和工单。模型有多租户边界，并可选取启用且版本最高的工艺模板。

可借鉴的是“产品与工艺版本分离”；不足是产品维度较浅，无法直接承担本项目约 20,000 SKU 的规格别名、产品族、补库算法、标准产能、模具和历史生效关系。

### 3.2 生产需求结构

OpenMES 主要把客户订单信息和计划数量放在 Work Order 上，没有本项目所需的“补库建议审核后形成、可拆分/合并排产且不能丢失”的独立需求池。其 backlog 是未排程工单集合，适合界面参考，不适合作为本项目需求台账。

### 3.3 生产工单结构

Work Order 保存工单号、产品、产线、计划数量、已产数量、优先级、交期、计划起止时间和工艺快照。下级 Batch 管目标、产出、报废、批次/工位；Batch Step 是可执行步骤。三层结构清楚地区分订单、批次和步骤执行。

本项目可采用“需求—计划任务—工序执行任务”的相似层次，但不能照搬类、字段、服务或接口。

### 3.4 工艺路线和工序任务结构

工艺模板含版本、步骤、质量检查、媒体和清单。创建工单时将启用模板固化为工艺快照，再从快照生成步骤任务；后续模板修改不会悄悄改变历史工单。步骤按顺序从 PENDING/READY 到 IN_PROGRESS/DONE，也允许可选步骤跳过，前序未完成时不能开始。

“模板版本 + 发布快照 + 执行实例”是本项目最值得吸收的设计。不同之处是本项目应保留关系型快照字段和来源版本号，避免仅用一个 JSON 承担所有查询。

### 3.5 周排产和甘特图交互

计划器提供日/周/小时/月视图、左侧待排池、按产线分组的时间轴、任务拖放和拉伸。后端检查同产线时间重叠，冲突返回明确结果，授权用户可确认强制排入；还支持逾期提示和轮询刷新。

本项目应借鉴待排池、拖拽预览、冲突确认、视图切换和任务详情抽屉，但冲突判定要扩展到工序、设备、模具、日能力和前序可用量，而不只是同一产线的时间重叠。

### 3.6 工单状态流转

状态包含待处理、接受、生产中、阻塞、暂停、完成、拒绝、取消。阻塞异常可以自动把工单置为 BLOCKED；产量达到计划量可自动完成；批次执行会驱动工单进入生产中。

本项目应把“业务动作”“派生进度状态”“异常阻塞标记”分开，避免一个状态枚举同时表达所有维度。

### 3.7 车间报工流程

操作者界面面向平板和 PWA：按工单查看步骤说明、媒体、清单和物料批次，用大按钮开始、完成或报告问题；必需文档/清单未完成时可禁止结束。顺序生产的提示很强，现场认知负担低。

这是本项目车间页面的主要交互参考，但本项目初期不做离线同步和复杂文档清单，先保证工序授权、数量、时间、未完成原因和异常提交。

### 3.8 计划、完成和剩余数量

OpenMES 用工单 `planned_qty` 与 `produced_qty` 判断完成，批次另记目标、产出和报废。这个口径适合单工单执行，却不足以说明同一需求被多个任务占用以及工序间重复报量。

本项目必须保留两个不同剩余量：

- `remaining_to_schedule = confirmed_qty - active_allocated_qty`
- `remaining_to_complete = confirmed_qty - qualified_completed_qty`

### 3.9 异常预警方式

Andon 异常按 Open → Acknowledged → Resolved → Closed 流转，可关联工单、步骤、物料和异常类型；阻塞型且未解决的异常会阻止步骤开始或工单推进。可记录数量、根因、围堵、处置和行动。

本项目应采用“异常是业务对象而非红色标签”的原则，并增加稳定去重键、持续天数、责任工序和系统建议动作。

### 3.10 用户权限结构

预置 Admin、Supervisor、Operator，并通过权限项控制管理、开始/完成步骤、创建异常等动作；操作者还受产线范围过滤。

本项目应将产线范围改为更贴合现状的工序/车间范围，并保留后端强制校验，不能只隐藏前端按钮。

### 3.11 数据库审计和修改记录

项目的审计模型记录对象、动作、修改前后值、用户、IP 和 User-Agent，README 强调 PostgreSQL 不可变审计。设计重点值得采用，但本项目还应把修改原因、请求 ID、计划版本和报工冲销写进统一审计链。

### 3.12 借鉴与不采用

借鉴：生产计划页面、车间操作页面、工艺发布快照、步骤前置约束、Andon 闭环、角色加产线范围。
不采用：AGPL 源码和样式实现、原技术栈、仅按产线判断容量、偏浅产品模型、用工单替代独立生产需求。

## 4. ERPNext

研究重点是官方文档中的 [Work Order](https://docs.frappe.io/erpnext/work-order)、[Job Card](https://docs.frappe.io/erpnext/job-card)、[Routing](https://docs.frappe.io/erpnext/routing)、[Workstation](https://docs.frappe.io/erpnext/workstation)和[Production Plan](https://docs.frappe.io/erpnext/production-plan)，并与当前 develop 源码核对。

### 4.1 产品主数据结构

ERPNext 的 Item 是完整 ERP 主档，含编码、名称、品类、库存单位及换算、变体、批次/序列、补货参数、采购销售属性、默认 BOM、仓库和估价设置。它证明产品身份、单位、变体和默认制造配置应分层，但大部分采购、销售、会计字段不属于本项目。

### 4.2 生产需求结构

Production Plan 可汇总销售订单、物料请求或手工产品行，形成待生产数量并批量生成 Work Order 和物料请求；来源链保留在计划和工单中。可借鉴“需求来源不能丢”和“计划只分配需求、不替代需求”。本项目仍以补库建议和少量人工需求为入口，不实现销售订单全流程。

### 4.3 生产工单结构

Work Order 关联产品、BOM、数量、计划日期、WIP/成品仓、所需物料、操作、Production Plan、销售订单/物料请求。状态由单据状态、物料转移、产出和工序进度综合派生，包括 Draft、Submitted、Not Started、In Process、Completed、Stopped、Closed、Cancelled 等。

本项目借鉴来源、操作快照、计划/实际和关闭/取消区分；不引入库存会计、领料、成品入库、分包和成本核算链。

### 4.4 工艺路线和工序任务结构

Routing 是可复用的工序序列模板；工序保存 Workstation、时间、批量与成本等。Work Order 提交后按操作生成 Job Card，Job Card 绑定工序、工作站、人员和时间日志；顺序 ID 可约束前置工序。

这套“路线模板—工单操作快照—Job Card 执行”的边界非常适合参考。本项目把 Job Card 的概念实现为 `operation_task`，字段和接口自行设计。

### 4.5 周排产和甘特图交互

ERPNext 更偏 Production Plan、工作站可用时间、能力规划和日历视图，核心操作仍有明显单据系统特征。它适合参考工作站日历、节假日、可用工时和替代工作站，不应成为本项目周排产界面的直接范本。

### 4.6 工单状态流转

ERPNext 把 Draft/Submit/Cancel 等单据生命周期与 Not Started/In Process/Completed 等执行进度组合，状态不是前端随意选择。停止、关闭和取消有不同语义。

本项目不照搬“提交型单据”，但应区分：草稿/发布属于计划生命周期；待产/执行/完成属于任务进度；暂停/取消/关闭属于业务动作。

### 4.7 车间报工流程

Job Card 支持开始、完成、员工分配、分段时间日志、完成量、待完成量和工序损耗；完成 Job Card 会更新 Work Order 操作进度。物料请求和转移可嵌入现场流程。

本项目首版只取开始/结束、人员、工作站、合格/不合格/报废和未完成原因；物料领退、工资计件和库存过账留在边界外。

### 4.8 计划、完成和剩余数量

ERPNext 区分工单数量、已生产、过程损耗和工序完成量，并有超产容差校验。Job Card 的待完成量随既有报工更新。可借鉴容差、工序量和最终合格产出的区分，但本项目完成需求只认最终合格量，前序工序报量不能重复增加需求完成量。

### 4.9 异常预警方式

通过业务校验、工作站时间冲突、缺料提示以及可配置 [Notification](https://docs.frappe.io/erpnext/notifications) 触发邮件/系统通知。灵活性很强，但配置型通知引擎对首版过重。

### 4.10 用户权限结构

[权限体系](https://docs.frappe.io/erpnext/permissions)以角色到 DocType 的读、写、创建、提交、取消、修改等权限为基础，还可用 permission level 控字段、用 User Permission 限关联对象。可借鉴“对象 + 动作 + 数据范围”三层模型，本项目不需要通用 DocType 权限引擎。

### 4.11 数据库审计和修改记录

[Document Versioning](https://docs.frappe.io/erpnext/document-versioning)记录被跟踪字段变化；可提交单据还可通过 [Audit Trail](https://docs.frappe.io/framework/user/en/audit-trail)串联取消和修订版本。可借鉴“修订不覆盖历史”，本项目用计划版本、追加写审计和报工冲销实现。

### 4.12 借鉴与不采用

借鉴：Work Order/Job Card/Routing/Workstation 的对象边界，来源追踪，工作日历，派生状态，取消与修订审计。
不采用：GPL 代码、Frappe DocType/Bench 技术体系、完整 ERP 物料与会计闭环、页面和字段的全量复制、过重的通用配置能力。

## 5. frePPLe

研究依据包括官方模型文档：[Sales orders](https://frepple.com/docs/current/model-reference/sales-orders.php)、[Manufacturing orders](https://frepple.com/docs/current/model-reference/manufacturing-orders.php)、[Resources](https://frepple.com/docs/current/model-reference/resources.html)、[Buffers](https://frepple.com/docs/current/model-reference/buffers.php)和 [Gantt plan editor](https://frepple.com/docs/current/a-day-in-the-life/production-planning/optimize-plan-in-gantt-chart.php)。

### 5.1 产品主数据结构

Item 支持层级、分类、成本、MTS/MTO 类型、重量体积、覆盖周期和单位。产品与地点、库存 Buffer、Operation 和 Demand 共同构成计划网络。对本项目有价值的是产品族层级、生产策略和单位；多地点分销网络暂不需要。

### 5.2 生产需求结构

Demand 含客户、产品、地点、交期、数量、优先级、最小发运、最大迟交、交付策略和批次。open/quote 才参与计划，历史 closed demand 可用于预测。需求与供应通过 pegging 关联，能回答“这个生产建议服务哪条需求”。

本项目应强制保存需求到任务的分配关系，并保留月度销售历史；不实现客户级交付策略和全网供需 pegging 求解。

### 5.3 生产工单结构

Manufacturing Order/OperationPlan 保存操作、起止、数量、完成量、批次和状态。proposed 是求解器建议，approved 可继续重排，confirmed 相当于冻结，completed 表示执行完成，closed 表示不再参与计划。

本项目可借鉴“建议计划—批准计划—冻结计划—执行完成”的分层，但用 `production_plan_version` 与 `production_task` 表达，不引入 frePPLe 求解对象。

### 5.4 工艺路线和工序任务结构

Operation 支持固定时间、按量时间、routing、alternate 和 split；可声明物料流、资源负荷、技能、批量、时间围栏、日历、最小/倍数/最大批量以及前后依赖。表达能力强，适合 APS。

本项目只吸收顺序、标准工时、设备/模具、能力日历、可选资源和生效期。分流比例、复杂替代网络和 setup matrix 不进入首版。

### 5.5 周排产和甘特图交互

frePPLe 的关键不是画条形，而是计划变更后继续校验物料、资源和前后关系。计划编辑器可拖动起止时间、切换替代操作/资源并更新计划。部分高级计划编辑、库存计划和企业功能受版本/商业许可限制，不能假定都属于 MIT 社区版；应对照[版本功能表](https://frepple.com/docs/current/features.html)逐项确认。

本项目首版采用“人工排产 + 实时规则校验”，不内嵌 APS 求解器；每次拖拽返回负荷、冲突、前序和交期影响。

### 5.6 工单状态流转

proposed/approved/confirmed 的价值是表达计划承诺强度，避免所有任务都被任意重排。completed/closed 又区分执行完成与计划不再考虑。

本项目在发布计划后可设置 `is_frozen` 或冻结窗口，插单若影响冻结任务必须显式授权并记录原因。

### 5.7 车间报工流程

frePPLe 核心是计划，不是轻型现场执行。社区版不能替代本项目的报工、异常、人员和工序授权；不能因为采用其计划思想而把现场流程依附在外部 APS 上。

### 5.8 计划、完成和剩余数量

它区分 operationplan quantity、quantity completed、库存投影、需求交付和 pegging 数量，并计算延迟、关键度和可行性。可借鉴“剩余不只是单点减法，还要结合时间和资源看风险”，但本项目的权威数量仍应来自需求分配和不可变报工汇总。

### 5.9 异常预警方式

[Problem report](https://frepple.com/docs/current/user-interface/plan-analysis/problem-report.php)把无效需求、资源超载、库存不足、操作前后冲突等统一为问题记录；[Inventory status](https://frepple.com/docs/current/user-interface/plan-analysis/inventory-status.php)用红黄绿表达风险，同时给出投影数量依据。

本项目应采用统一异常对象和数值证据，颜色只作为严重程度辅助，不能让用户猜原因。

### 5.10 用户权限结构

采用 Django 用户、组和模型权限，并可按场景控制访问。可借鉴组权限和计划场景隔离，但整个数据库复制式的 what-if scenario 对本项目过重，计划版本足够。

### 5.11 数据库审计和修改记录

数据维护界面提供变更历史，计划还能使用场景对比。其审计重点更偏输入数据与计划场景，不等同于 MES 现场追溯。本项目必须自行实现业务级前后值、修改原因和报工冲销审计。

### 5.12 借鉴与不采用

借鉴：历史需求预测输入、时间分段库存、补货目标、有限资源日历、冻结窗口、需求供应分配、可行性/延迟/关键度和标准化问题报告。
不采用：C++/Django 技术栈、完整 APS 求解器、多级分销网络、数据库整场景复制、许可未确认的企业版功能、把首版做成算法平台。

## 6. yuwang/MES

研究依据是其 [README 与分层说明](https://github.com/yuwang/MES/blob/876234f5131a9bb74f36e5c38c216583a8eb45d0/README.md)，以及该版本的领域实体、应用接口、基础设施服务、API 控制器和 React 页面。以下只描述行为，不复述源码。

### 6.1 产品主数据结构

Product 仅含编码、名称、规格和启停状态，并关联工单。结构易懂，但缺少单位、产品族、工艺、补库策略、别名、设备/模具能力等，不能直接支持本项目规模。

### 6.2 生产需求结构

没有独立生产需求或补库建议，工单直接人工创建。这会使“建议是否已审核、需求是否全部排产、为何产生工单”无法完整回答，本项目不能采用。

### 6.3 生产工单结构

WorkOrder 保存单号、产品、目标量、累计良品、累计不良品、状态、创建人和开始/完成时间；WorkReport 关联工单、工位、良品、不良品、人员和报工时间。模型小，适合作为轻量分层示例。

### 6.4 工艺路线和工序任务结构

未实现 Routing 或工序任务；PRD 只预留工艺路线接口。报工直接挂工单和工位，无法表达多工序顺序、前序量、任务分派和各工序进度。

### 6.5 周排产和甘特图交互

未实现周排产或 Gantt。工单页面是标准筛选、分页、表格和新增/编辑弹窗。因此它不提供排产设计参考，只能证明轻型 CRUD 可保持简单分层。

### 6.6 工单状态流转

只有 Pending、InProgress、Completed、Cancelled。首次报工自动进入生产中，累计良品达到目标自动完成；待产或生产中可取消。简单直观，但修改/删除报工后未完整恢复已经完成的状态，说明派生状态必须集中重算。

### 6.7 车间报工流程

操作者从生产中工单和启用工位中选择，输入良品、不良品和备注；管理员/操作者可修改报工，管理员可删除。这个流程上手快，但直接改删历史报工不符合制造追溯要求，也缺少任务、班次、时间段、报废和异常原因。

### 6.8 计划、完成和剩余数量

新增报工时对工单累计量做增量加法，修改时计算差额，删除时减回；良品达到目标则完工。该做法在并发、部分失败、状态回退和审计方面风险较大。可借鉴数量分类的直观展示，不采用可编辑/删除报工与手工维护汇总值的方式。

### 6.9 异常预警方式

有设备实时报警与看板统计，但业务异常体系尚未形成。未来规划才包含质量异常预警。不能满足漏排、少排、前序不足、连续未排和计划逾期等需求。

### 6.10 用户权限结构

角色包含 Admin、Planner、Technician、Operator，角色实体有 JSON 权限声明；API 主要通过固定角色名限制控制器动作。JSON 权限并未形成完整动态授权和数据范围模型。

本项目可保留四类初始角色，但数据库权限码必须由统一授权服务执行，并叠加工序/车间范围。

### 6.11 数据库审计和修改记录

存在 OperationLog 实体，记录用户、操作类型、描述和 IP，但未看到覆盖关键服务的完整写入链，也没有 before/after、对象标识、原因和防修改设计。BaseEntity 只有创建/更新时间，不能视为审计。

### 6.12 借鉴与不采用

借鉴：项目按 API、Application、Domain、Infrastructure、Shared 分层的清晰职责；DTO、服务接口和仓储边界；工单/报工页面的低学习成本。
不采用：.NET/React/MySQL 技术栈、工单即需求、无工序任务、增量维护累计量、直接修改/删除报工、固定角色名散落在控制器、未落地审计。许可证正文缺失也要求禁止复制代码。

## 7. 对本项目的设计约束

1. 独立保存建议、需求、分配、计划任务、工序任务和报工，不用一个“工单”包揽全链路。
2. 工艺采用版本化模板；计划发布时固化快照，历史执行不随主数据漂移。
3. 周排产首版是人工决策支持：服务端计算冲突和影响，不承诺 APS 自动最优解。
4. 计划数量、完成数量和剩余数量只定义一套权威公式，汇总字段可缓存但必须可重算。
5. 报工追加写，错误用冲销与更正记录解决；禁止物理删除已生效报工。
6. 异常必须有对象、规则、数值证据、严重度、状态、责任和解决记录。
7. 权限采用角色、动作权限和数据范围三层；审计与业务更新同事务，计划另有版本链。
8. 所有实现保持既定技术栈，参考项目不作为运行依赖，也不引入其框架。
