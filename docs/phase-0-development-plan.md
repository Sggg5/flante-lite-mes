# 阶段 0：分阶段开发计划与第一阶段验收清单

## 1. 开发阶段

每个阶段都以“目标、修改范围、验收方式”开始，以“已完成、未完成、测试结果、下一阶段建议”结束。数据库结构变化统一通过 Alembic，关键计算必须有单元测试。

### 阶段 1：工程骨架、权限骨架与样表解析

目标：建立可运行、可迁移、可测试的前后端工程，并证明两份样表能够被安全识别和预览。

范围：

- `frontend`：Vue 3、TypeScript、Vite、Element Plus、Pinia、Vue Router、Vitest。
- `backend`：Python 3.12、FastAPI、SQLAlchemy 2、Pydantic、Alembic、pytest。
- SQLite 开发配置、PostgreSQL 生产配置、Docker Compose。
- 用户、角色、用户角色、审计日志的最小表结构与登录骨架。
- 导入批次、工作表元数据、行错误表结构。
- Excel 元数据扫描：工作表、真实范围、多行表头候选、合并区域、公式、外部链接、预览。
- README 的 Windows 与 Docker 启动步骤。

明确不包含：补库正式计算、需求池、排产、报工和完整业务页面。

### 阶段 2：产品主数据与导入中心

目标：将销售、库存、两类在制、常规产品安全导入可追溯快照。

范围：产品/产品族/别名/补库策略；上传、选表、表头和字段映射、预览、校验、错误下载、提交、重复防护和受控撤销；服务端产品搜索分页。

验收重点：产品编码按文本；百万空行不拖垮导入；负数在制保留原值并生成异常；重复提交幂等。

### 阶段 3：补库计算与审核

目标：实现六种目标库存算法、计算快照、批量审核和转生产需求。

范围：月度销售聚合、库存/在制选择、已排未开工扣除、补库计算批次、建议中心、人工确认、修改原因、审计日志。

验收重点：全部算法边界测试；输入快照可还原；建议小于等于 0 不生成需求；同建议重复转化不重复建需求。

### 阶段 4：生产需求池与工艺主数据

目标：建立不可丢失的需求生命周期和工序模板。

范围：需求状态、剩余待排/待完成、取消/关闭；工序、模板、模板步骤、设备、模具、产能；确认未排、少排、跨周未排、负在制和数据缺失规则。

验收重点：部分排产后剩余量继续可见；跨周不消失；取消有完整审计；20,000 SKU 搜索性能可接受。

### 阶段 5：周排产中心

目标：将需求可靠分配到工序、设备和日期，支持拆分、合并、插单、顺延、版本和 Excel 导出。

范围：周计划、版本、任务、每日计划、需求分配、产能/设备/模具/前序校验；四工序近原格式导出。

验收重点：并发分配不超量；发布前规则齐全；插单后原任务有新日期或明确异常；导出后计划/实际双行布局正确。

### 阶段 6：报工、结转与完整异常闭环

目标：班组长可按工序报工，未完成量明确结转，需求完成量自动更新。

范围：任务工作台、报工/冲销、合格/不合格/报废、未完成原因、结转；无报工、逾期、完成不足未结转等规则。

验收重点：报工冲销可追溯；未完成不自动视为完成；权限限制到工序；需求完成状态准确。

### 阶段 7：看板、性能、生产化部署

目标：完成业务看板、全链路审计、PostgreSQL/Docker 部署和生产前质量验证。

范围：看板指标和 ECharts、查询优化、任务队列/定时扫描、备份恢复、日志与监控、安全配置、数据迁移演练。

验收重点：关键列表、20,000 SKU 搜索、大批量导入和异常扫描达到约定性能；备份恢复和回滚演练通过。

## 2. 阶段 1 验收测试清单

### 2.1 工程与启动

- [ ] Windows 按 README 可分别启动前端和后端。
- [ ] `docker compose up` 可启动前端、后端和 PostgreSQL；健康检查通过。
- [ ] Python 版本锁定 3.12，Node 版本在 README 和配置中明确。
- [ ] FastAPI `/health` 返回服务、数据库和版本状态。
- [ ] Vue 初始页可调用后端健康接口，并正确显示成功/错误状态。
- [ ] 开发环境默认 SQLite，生产配置切换 PostgreSQL 时无需修改业务代码。

### 2.2 数据库与迁移

- [ ] 全新数据库执行 `alembic upgrade head` 成功。
- [ ] 执行降级再升级成功，不丢迁移链。
- [ ] 阶段 1 所有表均有 `created_at`、`updated_at`。
- [ ] 用户名、角色编码、导入批次号和文件去重键有数据库唯一约束。
- [ ] 时间字段和数量字段类型符合设计；数量不使用浮点列。

### 2.3 权限与审计骨架

- [ ] 初始化管理员可登录；错误密码返回统一错误结构且不泄露账号是否存在。
- [ ] ADMIN、PLANNER、FOREMAN、VIEWER 四角色可初始化。
- [ ] 未认证访问受保护接口返回 401，无权限返回 403。
- [ ] 修改用户角色写入审计日志，包含操作人、前值、后值、时间和请求编号。
- [ ] 密码只保存安全哈希，不进入日志和 API 响应。

### 2.4 Excel 元数据扫描

对两份真实样表逐项验收：

- [ ] 正确识别第一份工作簿 7 张表和第二份工作簿 4 张表。
- [ ] “常规排产产品”真实范围识别到 2,211 行附近，不遍历/展示 1,039,600 个空格式行。
- [ ] “最终整理版本”真实范围识别到 506 行附近，不遍历/展示 1,048,087 个空格式行。
- [ ] 能识别补库表无合并单元格，以及周计划四表的大量合并区域。
- [ ] 能识别周计划的计划/实际双行结构。
- [ ] 能识别“制管”“成型”公式中的外部工作簿引用，并显示外部链接风险。
- [ ] 能预览公式文本和缓存值，但不把缓存值当作权威计算结果。
- [ ] 能识别中文文本、日期序号、整数日号、空白列和多行表头。
- [ ] 预览默认限制行数，不因 26.8 万行销售明细阻塞请求。

### 2.5 字段规范化与校验原型

- [ ] 产品编码 `000002412067` 保留前导零。
- [ ] 字母数字混合编码和连字符编码保持原样。
- [ ] 数值/科学计数法编码不会未经确认直接写入产品主键。
- [ ] 文本 `"0"`、数值 0、空白三者原始值可区分，规范化结果符合规则。
- [ ] 小数数量使用 `Decimal`，无二进制浮点误差。
- [ ] 负在制被识别为警告/错误候选，原始负值不被静默改写。
- [ ] 周计划合并单元格中的设备、产品和批次只在合并覆盖范围内向下继承。
- [ ] 缺产品编码的周计划行进入待匹配状态，不按名称自动提交。

### 2.6 API 错误处理与幂等

- [ ] 文件不存在、扩展名错误、损坏工作簿、加密工作簿和超限文件返回明确错误。
- [ ] 不支持的公式/扩展不被吞掉，至少形成可查看的警告。
- [ ] 相同文件、相同导入类型和来源日期重复上传时给出重复提示。
- [ ] 上传中断或解析失败不会留下“已完成”批次。
- [ ] 所有错误响应含稳定错误码和 `request_id`；服务端日志可按其追踪。

### 2.7 自动化测试与质量门槛

- [ ] `pytest` 全部通过，覆盖 Excel 真实范围、合并填充、编码规范化、日期解析、负数和重复文件。
- [ ] `vitest` 全部通过，覆盖导入向导状态与 API 错误展示的基础组件。
- [ ] TypeScript 类型检查通过。
- [ ] Alembic 迁移测试通过。
- [ ] 不通过删除测试、关闭类型检查或捕获后忽略异常来获得绿色结果。
- [ ] CI 或本地统一命令可一次运行后端测试、前端测试和类型检查。

## 4. 进入阶段 1 前需要人工确认的决策

这些问题不影响阶段 0 结论，但会影响阶段 2/3 的最终实现：

1. 销售负数量应全部作为退货冲减，还是需按出库类别分别处理。
2. 目标库存数量是否需要按最小生产批量向上取整，以及取整规则。
3. 六个月加权平均的默认权重；建议从近到远为可配置权重并校验合计。
4. 周计划中缺产品编码的批次，能否从 ERP 提供“批次号 → 产品编码”对照表。
5. “包装”表中同一批次 `20260701` 对应多个产品，是正式批次规则还是临时汇总编号。
6. 下料计划周期与其他工序不一致是否为常态；系统应允许每工序独立周界限，还是统一自然周。
7. 在制负数是盘点调整、已超额入库还是数据错误；系统会保留原值并按 0 参与补库，但需确定责任流程。
8. 第一版 Excel 导出要保留哪些颜色、合并布局、备注列和外部日报公式；建议只导出静态计划，不继续生成本地路径外链。

人工确认阶段 0 后，再开始阶段 1；本次提交不包含阶段 1 功能代码。
