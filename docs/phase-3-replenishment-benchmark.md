# 阶段 3 补库计算性能基准

## 基准环境和数据

2026-07-18 在 Windows 本地开发环境、SQLite 数据库上实际运行：

```powershell
cd backend
.\.venv\Scripts\python.exe scripts\benchmark_replenishment.py `
  --shipments 268000 --products 12000 --regular-products 2210
```

脚本动态生成完全虚拟的产品编码、销售日期、数量、库存和在制记录，不读取或保存真实公司 Excel/经营数据。销售明细按块写入；计算阶段由数据库完成六个月销售聚合，并批量加载产品、策略、库存和在制快照。

## 实测结果

| 指标 | 结果 |
|---|---:|
| 销售明细 | 268,000 |
| 产品目录 | 12,000 |
| 常规排产产品 | 2,210 |
| 计算快照 | 2,210 |
| 正数建议 | 2,210 |
| 转换需求 | 2,210 |
| 重复需求 | 0 |
| 虚拟数据写入 | 13.72 s |
| 补库计算 | 1.45 s |
| 结果查询 | 0.14 s |
| 批量审核 | 0.11 s |
| 转生产需求 | 0.60 s |
| 总耗时 | 17.91 s |
| 峰值常驻内存 | 125.47 MB |
| SQLite 文件大小 | 79.06 MB |

## 结论

2,210 个常规产品均生成计算快照，268,000 条销售记录未加载为 Python 明细列表，转换后未产生重复需求。基准重点证明了数据规模和查询形态；生产 PostgreSQL 的实际耗时仍取决于磁盘、连接池、并发、VACUUM/统计信息和部署资源，应在上线环境继续建立周期性基准。
